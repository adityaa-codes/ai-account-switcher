# v0.3.0 Release Notes

**Release date:** 2026-03-10

## Highlights

v0.3.0 is a stability, resilience, and isolation release. It fixes a long-standing 400 error from the Gemini quota API, hardens both hooks against a breaking change in Gemini CLI ≥ 0.33, adds per-profile isolation for Codex, expands pool management with four new sub-commands, and improves day-to-day observability with discrete log files and new diagnostic commands. Test suite grows to 471 tests.

---

## Bug Fixes

### Health command — fixed HTTP 400 errors

The `loadCodeAssist` API requires a specific metadata payload. The previous implementation sent `{"supportedFeatures": ["GEMINI_CLI"]}`, which the API rejected with HTTP 400. The correct body is now sent:

```json
{
  "metadata": {
    "ideType": "GEMINI_CLI",
    "platform": "PLATFORM_UNSPECIFIED",
    "pluginType": "GEMINI"
  },
  "mode": "HEALTH_CHECK"
}
```

The `mode: HEALTH_CHECK` field also prevents billing side-effects during health validation.

### Idempotent hook installation

`switcher install` previously appended duplicate hook entries to `~/.gemini/settings.json` on each run. Hook installation now updates entries in-place and removes stale entries for unregistered hooks. Re-running `switcher install` is now always safe.

### Portable date formatting

`_format_reset_date()` used the `%-d` `strftime` format code (GNU libc only), which raised errors on non-glibc Linux systems. Replaced with `str(dt.day)` for full portability.

## New Features

### 🛡️ Hook safety — `stopHookActive` support

Gemini CLI ≥ 0.33 sets `stopHookActive: true` in hook input JSON when the agent loop is stopped by the user. Both `gemini_before_agent` and `gemini_after_agent` now bail out with `{}` immediately when this flag is detected, preventing the hook from issuing API calls or triggering rotations into a dead session.

The `AfterAgent` retry response now also explicitly includes `"stopHookActive": false` to prevent the flag from propagating to the next turn.

### 🔑 OAuth client discovery and caching

The OAuth client credentials needed to call Google's token API are now discovered automatically from the local `gemini-cli-core` package, trying three candidate paths:

1. `dist/src/code_assist/oauth2.js`
2. `dist/src/auth/oauth2.js`
3. `dist/src/oauth.js`

Discovered credentials are cached at `~/.config/ai-account-switcher/cache/oauth_client.json` with a 24-hour TTL, eliminating repeated filesystem scans on every health check. On cache miss, a warning is logged suggesting the user run `gemini` once to refresh.

### 📦 Pool sub-commands

Four new sub-commands under `switcher gemini pool` (and `switcher codex pool`):

| Command | What it does |
|---|---|
| `switcher gemini pool list` | Lists all profiles in the rotation pool |
| `switcher gemini pool health` | Runs health checks across the pool |
| `switcher gemini pool export [--dest DIR]` | Exports all pool profiles to a directory |
| `switcher gemini pool status` | Shows rotation state and the next-up profile |

### 🧠 Codex per-profile isolation

When switching Codex profiles, the following are now automatically snapshotted from the departing profile and restored for the incoming profile:

- **Conversation memory** (`codex_memory.py`) — SQLite DB and flat-file memory store, saved per profile under `~/.config/ai-account-switcher/profiles/codex/<label>/`
- **Plugin list** (`codex_plugins.py`) — installed plugins are recorded per profile; a warning is printed when two profiles have diverging plugins
- **Sandbox policy** (`codex_sandbox.py`) — `policy.toml` is saved per profile and restored on switch

All three are integrated into `CodexProfileManager.switch_to()` with no user action required.

### 📝 Per-profile system prompt (Gemini)

Add a `system_md` key to a Gemini profile's `meta.json` and `env.sh` will automatically export `GEMINI_SYSTEM_MD` when you switch to that profile:

```json
// ~/.config/ai-account-switcher/profiles/gemini/work/meta.json
{
  "label": "work",
  "system_md": "/home/you/prompts/work-system.md"
}
```

Set `"write_system_md": true` to also export `GEMINI_WRITE_SYSTEM_MD=1`.

### 📊 Quota display — used % instead of remaining %

The progress bars in `switcher gemini quota` and `switcher gemini health` now show **quota used** rather than quota remaining. A fuller bar now means more quota has been consumed — the more intuitive framing for capacity planning.

Additionally, the profile's quota tier name (e.g. `Standard`, `Gemini Code Assist Enterprise`) is displayed when returned by the API.

### 🔔 Alerts command

```bash
switcher alerts          # last 20 error-log lines (default)
switcher alerts --lines 50
```

Tails `~/.config/ai-account-switcher/logs/errors.log` directly from the CLI — no need to locate the file manually.

### 🔍 Version update check

```bash
switcher version --check
```

Fetches the PyPI JSON API and reports whether a newer version of `cli-switcher` is available.

### 📁 Discrete log files

Two new log files complement the existing `switcher.log`:

| File | Contents |
|---|---|
| `logs/errors.log` | ERROR-level entries only — the fast triage file |
| `logs/commands.log` | One line per command: name, elapsed ms, exit status (OK / ERROR / INTERRUPT) |

## Improvements

### API tier and timestamp fields

`ProfileQuotaInfo` now exposes:
- `tier: str | None` — populated from `currentTier.tierName` in the quota API response
- Unix epoch integer `reset_at` values are normalised to ISO-8601 strings before display
